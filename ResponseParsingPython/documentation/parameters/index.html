Validation Parameters

This document describes the validation parameters that are used in Nagwa content.

Validation parameters are a set of parameters that determine how a student must write their answer for a given question.

Validation parameters often deal with different ways of writing the same thing. For example, '23' and '023' are different ways of writing the same number, and we may want to accept both of them as the right answer to a question.

Different parameter values can be set for each question. While most of the time you will probably want to use the default value of each parameter, sometimes you will want to change them.

This document describes what the different values are, and what effect they have.

The primary answer types

Each question expects an answer of a certain primary type.

The primary type of an answer is what kind of mathematical expression it is. Some examples are: an integer, a decimal, a fraction, a surd, a complex number, an inequality, a single word, et c.

At the moment, there are validation parameters for four primary types, but more primary types will be added in future.

Integer

Any whole number between negative infinity and positive infinity.

42
+15
-300
2500
1000000

Non-negative Integer

Any whole number between zero and positive infinity.

42
+15
300
2500
1000000

Decimal

Any rational number between negative infinity and positive infinity.

12.5
3.400
1.785
3.14159
-2400.0078

Currency Value

Mostly the same as the decimal type, but a few extra limitations are placed on it.

See here for more about this type. →

The primary type affects what parameters can be set for a question. For example, a parameter that can be set for the decimal type is 'Must have exactly n decimal places', which requires the student to give an answer that has n decimal places, but this parameter obviously can't be set for the integer type, because integers have no decimal places.

NOTE: There are many questions in our content that have answers like '9.6 km/s' or '12 mph' (i.e., a number with a unit after it). This is common in mechanics content in maths, and of course throughout physics. Although these keys have integers and decimals as part of them, they should not be set to the integer or decimal answer types. The integer and decimal answer types are only for questions where the answer is an integer or decimal ONLY. Anything else must have the answer type 'none'. This validation system has so far only been designed to cover a small number of maths questions. The majority of the questions in our content will (for now) have an answer type of 'none', as they are not covered by the validation code.

What are leading zeros?

A 'leading zero' is any '0' digit that appears at the start of a number.

For example

0023 has two leading zeros
00023 has three leading zeros
23 has no leading zeros

All of the above are valid ways of writing the number 23. Most of the time we would never write '0023' or '00023', but all of the above technically represent the number 23. If the answer to a question were 23, but the student gives their answer as '0023', they are still correct.

↑ Back to the table of contents

There are two parameters for how to handle leading zeros

'Allow Leading Zeros'

'Remove Leading Zeros'
– can be true or false – can be true or false
– set to false by default – set to false by default

This parameter determines whether or not a student can submit an answer that has leading zeros.

If it is set to false for a question, then if a student tries to submit an answer to that question like '0023', they will not be allowed to submit the answer, and will be given the message 'You don't need to write zeros at the start of your answer.'

If it is set to true, then a student can submit answers like '023', '0023', '00023', and all will be allowed through.

Imagine that a student enters '0023' as their answer to a question. (And that 'Allow Leading Zeros' has been set to true for that question.)

If 'Remove Leading Zeros' is set to true, then this answer will be converted into '23' before it is compared to the key.

This means that even if a student does put leading zeros in their answer, it doesn't matter, because we're going to remove them before checking their answer.

If this is set to false, leading zeros won't be removed from the student's answer before it is checked.

Common settings

The most common set of values that you are likely to want to use for the above two parameters are

'Allow Leading Zeros' true
'Remove Leading Zeros' true

This is the "friendliest" or "nicest" setting. A student can type leading zeros in their answer (if they really want to), but they will automatically be removed before their answer is checked. This way, if a student types '0023', but the key is '23', they won't be "told off" and they won't get the question wrong.

Sometimes, however, you might want to encourage the student not to write leading zeros. (This would be particularly useful at lower g-values, because we want to encourage young students to write numbers in the conventional way.) In this case, you could set 'Allow Leading Zeros' to false - then if they try to type '0023', they will be told not to write the leading zeros.

An exception ...

There is one case where most of the time we do want a student to write a leading zero, and that's with decimals between −1 and 1.

0.1
0.55
0.4298

All of the above numbers have one leading zero, and this is generally the preferred way of writing decimals between −1 and 1.

Even if 'Allow Leading Zeros' is set to false, students will still be allowed to submit decimals of the form '0._____'. (The leading zero is thought of as a 'phantom' zero.)

However, there is a parameter to control whether this zero will be added to a student's answer (if they haven't written it), before their answer is checked against the key.

'Add a single leading zero to decimals smaller than 1'

This parameter can take values of true or false.

It is set to true by default.

If set to true, when a student enters something like '.12' as their answer, it will automatically have a zero added to the front of it, making it '0.12'.

If set to false, when a student enters something like '.12' as their answer, no leading zero will be added to it.

What are trailing zeros?

A 'trailing zero' is any '0' digit that appears at the end of a number and after the decimal point.

For example

1.500 has two trailing zeros
1.5000 has three trailing zeros
5.0000 has four trailing zeros

But note that

1.00500 has only two trailing zeros
200.40 has only one trailing zero
200.0040 has only one trailing zero
23000 has no trailing zeros

When numbers are considered to be exact, by convention we don't normally write out any trailing zeros to a number (because the trailing zeros technically go on forever).

Note that while the leading zeros parameters can be applied to all four of the current primary types, the trailing zeros parameters can only be applied to the decimal type. (Integers don't have trailing zeros.)

An odd case ...

What happens if a student types in something like '12.' as their answer - i.e., a decimal with no numbers after the decimal point?

An answer like this is considered a valid decimal. But perhaps the key is just '12' - in this case neither the 'Allow Trailing Zeros' nor the 'Remove Trailing Zeros' will have any effect on the student's answer - because there are no trailing zeros.

There is a special parameter for removing this trailing decimal point before the student's answer is compared to the key. This parameter is called 'Remove Trailing Decimal Point'. It can take a value of true or false. If true, '12.' would be changed to '12'. If false, '12.' would not be changed. It is set to true by default.

It would be very unusual to ever need to change this parameter - most of the time we don't want students to write something like '12.' (there are only a few cases where we might).

Plus and minus signs

When we write negative numbers, we always write a minus sign at the start of the number to indicate that it's negative.

When we write positive numbers, however, by convention we usually don't write a plus sign at the start of the number to indicate that it's positive. If a number is written without a plus or a minus sign, it's taken to be positive.

However, writing a positive number with a plus sign is still a valid way of writing a number, even if it's not how we normally write numbers.

Sometimes we may want to encourage a student not to write a plus sign at the start of a positive number (because we want to encourage them to follow the conventional rules for writing numbers). Other times, we may want them to explicitly write the plus sign (for example, when giving relative charges of particles in physics). And sometimes we may want the allow the student to do either, and have their answer marked correct regardless of whether or not it has the plus sign.

There are two parameters for how to handle signs

'Sign'

'Normalize Sign'
– can have a value of must be explicit, must be implicit, or can be explicit or implicit – can have a value of make explicit, make implicit, or do nothing
– set to can be explicit or implicit by default – set to do nothing by default

This parameter determines how a student is allowed to write their answer.

If it's set to must be explicit, and if a student gives a positive number as their answer to a question, then it must have a plus sign at the start of it. If it doesn't, they will be given a message telling them to add a plus sign.

If it's set to must be implicit, and if a student gives a positive number as their answer to a question, then it must not have a plus sign at the start of it. If it does, they will be given a message telling them to remove it.

If it's set to can be explicit or implicit, then a student will be allowed to submit a positive-number answer with or without a plus sign.

If the student gives a negative-number answer, this parameter has no effect.

This parameter determines whether or not a plus sign will be added to or removed from a student's answer before it is checked against the key.

If it's set to make explicit, and a student submits a positive-number answer without a plus sign, then a plus sign will be added before the answer is checked.

If it's set to make implicit, and a student submits a positive-number answer with a plus sign, then the plus sign will be removed before the answer is checked.

If it's set to do nothing, no plus sign will be added to or removed from the student's answer before it is checked.

If the student gives a negative-number answer, this parameter has no effect.

Conflicts

For any given question, it is possible to set more than one of the above three parameters.

This can create conflicts. For example, 'Must have at least n decimal places' could be set to '4', while 'Must have exactly n decimal places' is set to '2'. In such a scenario, both constraints would be applied, meaning that the student would not be able to answer the question. It is therefore very important to avoid accidentally choosing values like this.

Currency Values

The currency value primary type is a 'special case' of the decimal primary type.

For each country and each currency, there are usually conventions for how to write values of that currency. For example, in the UK, we generally don't write '£12.5' or '£12.500' - we would always write '£12.50'. So the currency value type is like the decimal type, but with some extra limitations.

The main limitations are around the number of decimal places that must be given for a currency value. For example, if the currency is set to GBP (UK pounds), any answer must be given to zero or two decimal places - if the student tries to submit anything else, they will not be able to submit their answer.

Which currencies are supported?

The table below shows which currencies are currently supported, and what limitations apply to them.

USD United States Dollars Answers must be written to zero or two decimal places.
GBP United Kingdom Pounds Answers must be written to zero or two decimal places.
EGP Egyptian Pounds Answers must be written to zero or two decimal places.
SAR Saudi Riyal Answers must be written to zero or two decimal places.

Parameters that can't be applied to currency values

There are several parameters that can't be applied to currency values.

'Must have at least n decimal places', 'Must have no more than n decimal places', and 'Must have exactly n decimal places' cannot be applied, as these would conflict with the in-built decimal place constraints for each currency. If these parameters are set for a currency value question, they will be ignored.

The significant figure parameters also can't be applied for the same reason.

And the two trailing zero parameters cannot be applied. This is because something like '£12.50', even though it has a trailing zero, is always a valid way of writing this currency value.
